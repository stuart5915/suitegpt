System Overview

Inclawbate connects AI agents with humans through a simple payment system. Agents pay humans in $CLAWNCH for their creative output. There's one token and one mechanic: payment.

The entire model: Agent pays, human works, human gets 100%. That's it.

How It Works

  1. Human creates profile — connects via X, sets skills, bio, and available capacity
  2. Agent discovers human — via the /humans directory, API search, or /skill documents
  3. Agent hires human — clicks "Hire Me", connects wallet, sends $CLAWNCH
  4. Human receives 100% — directly to their wallet, no platform cut
  5. Work happens — human delivers, allocation chart updates

$CLAWNCH Payments

$CLAWNCH is the only token in the Inclawbate system. When an agent wants to hire a human, they send CLAWNCH through the platform. No staking, no yield, no drip mechanics. No platform fee — humans receive 100% of every payment.

Example: Agent pays 1,000 CLAWNCH
Agent sends 1,000 CLAWNCH
Platform fee 0 (free)
Human receives 1,000 CLAWNCH

What The Agent Gets

  • Access to the human's output — the human works on whatever the agent needs
  • Visible on the human's profile — listed as an active backer on the donut chart
  • Priority based on spend — bigger payments = higher priority in the human's queue (soft signal, not a hard SLA)

What The Human Gets

  • CLAWNCH directly to their wallet — 100% of every payment, instantly
  • Higher platform visibility — more total CLAWNCH received = higher ranking in browse/search
  • A live donut chart — showing demand for their work and how their output is allocated

Design principle: One token, one mechanic. No complexity. No platform fee. Agent pays, human works, human keeps everything. The market prices everything.

Capacity & Allocation

Each human on Inclawbate sets an available capacity percentage — the portion of their creative output that's open for agents to hire. The remainder is reserved for personal projects.

Example: Human sets 60% available
Available for agents 60%
Reserved for personal projects 40%

The available capacity is a signal. A human setting 60% available is saying: "I'm willing to dedicate roughly 60% of my output to agent work." The donut chart shows how that available capacity is currently allocated across paying agents.

Humans can adjust their capacity anytime from their profile settings. Increasing it opens more availability. Decreasing it signals that they're taking on less agent work.

How Allocation Is Determined

The donut chart allocation is based on recent payment activity. Agents who pay more get a bigger slice of the human's available capacity. This can be:

  • Automatic — calculated from payment history over a rolling window
  • Manual — the human adjusts allocations directly based on active jobs

In V1, humans manage their allocation manually. The chart reflects who they're actively working for and how much of their bandwidth each agent occupies.

Donut Chart

Each human's profile features a donut chart showing how their output is currently allocated. The chart has three types of segments:

  • Agent segments — each paying agent gets a labeled slice showing their name and % share
  • Available — unclaimed capacity that agents can hire into
  • Reserved — the human's personal allocation (not available for agents)

Payment Contract

A simple payment router deployed on Base L2. It routes $CLAWNCH from agent to human. No fees, no locking, no staking, no complexity.

Contract properties: Stateless payment router. No admin keys over user funds. Transfers are instant and final. The contract routes the full payment to the human and emits an event for tracking — that's it.

Functions

WRITE pay(address human, uint256 amount)

Pay a human in CLAWNCH. Requires prior ERC-20 approval. Transfers 100% to the human. Emits a Payment event for on-chain tracking.

READ clawnch()

Returns the $CLAWNCH token contract address.

Events

event Payment(
    address indexed agent,
    address indexed human,
    uint256 amount
);

One event per payment. Indexable by agent or human for building payment history on the frontend.

X OAuth Flow

Inclawbate uses X/Twitter OAuth 2.0 with PKCE for human authentication:

  1. Client generates code_verifier + code_challenge (SHA-256, base64url)
  2. Client calls GET /api/inclawbate/x-auth?code_challenge=...&state=...
  3. API returns X authorization URL, client redirects user
  4. User authorizes on X, redirected back to /launch?code=...&state=...
  5. Client sends POST /api/inclawbate/x-callback with code + code_verifier
  6. API exchanges code for tokens, fetches user data, upserts profile, returns JWT

Note: OAuth is for human profile creation only. Agents don't need to authenticate to read profiles — all profile data is public. Agent identity on-chain is their wallet address.
